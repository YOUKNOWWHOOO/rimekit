## 1. 目录定位

本目录用于承载 Android 客户端正式工程。

## 1.1 验证联动

本文档覆盖范围内的正式要求，默认由以下几类逐条验证项共同承接：

1. `doc_baseline-*`
   - 用于验证目录说明、白名单边界、历史存量口径和冻结规则传播。
2. `platform_implementation-android-*`
   - 用于验证 Android 技术路线、平台职责、手动步骤、安装入口和平台编辑边界。
3. `end_to_end-*`
   - 用于验证 Android 用户路径进入真实导入、部署、回检、回滚和诊断闭环后的可用性。

使用规则固定如下：

1. 本文件新增、删减或改义了 Android 技术路线、目录边界、平台职责、安装入口、手动步骤或历史存量处理口径时，必须同步更新 `docs/需求验证映射.md`、`docs/测试与验收矩阵.md`、`docs/平台集成规范.md`、`docs/用户手册.md` 与 `docs/需求变更记录.md`。
2. 本文件对应证据默认落到 `/diagnostics/doc-baseline/requirements/`、`/diagnostics/platform-implementation/android/requirements/` 或 `/diagnostics/end-to-end/requirements/`。
3. 本文件中的目录规则、安装入口、设置跳转、导入源目录、平台编辑边界和历史存量限制都必须能在逐条验证动作中点名到具体目录、页面、按钮或平台结果对象。

开发启动前，本目录允许保留现有 Android 工程文件，但这些文件必须先按冻结文档审计分类；只有与正式主线冲突的演示骨架、重复实现或无效占位页面才应清理。

本目录的内容同样受 [需求规范](../../docs/需求规范.md) 与 [文档治理与冻结规范](../../docs/文档治理与冻结规范.md) 约束。

本说明文档是派生文档，不具备高于 `docs/` 的裁决权。

## 2. 正式技术路线

Android 客户端固定采用：

1. `Kotlin`
2. Android 原生工程
3. `Jetpack Compose`
4. `ViewModel + StateFlow`

进入开发后，应继续以现有 Android 工程为基础修改、更新和增加；只有与正式主线冲突的对象才应删除。

## 3. 正式职责

Android 客户端必须承担：

1. 检测 Android 承载器状态、输入法启用状态与当前输入法选择状态
2. 检测目录授权状态
3. 引导用户完成目录授权
4. 尽量直接发起承载器安装入口、输入法设置入口和输入法选择器，而不是只给静态说明
5. 引导用户完成输入法启用与切换，并在返回后自动回检
6. 读取和保存配置模型
7. 生成 Android 目标包、`t9.custom.yaml` 与 Android 应用清单
8. 执行或引导执行导入与部署
9. 参与跨端同步与恢复
10. 回检运行状态
11. 输出结构化诊断结果

Android 图形界面编辑边界固定如下：

1. Android 侧必须编辑全部共享字段与 Android 专属字段。
2. Windows 专属字段在 Android 侧只允许只读展示、同步结果展示和跳转提示。
3. Android 保存配置时必须保留 Windows 专属字段原值，不得清空、重置或重写。

正式工程启动后，本目录至少应能落出以下职责分层：

1. 页面层
2. Android 应用服务层
3. 平台适配层
4. 与共享契约层的接入层

## 4. 平台边界

Android 端允许存在系统不可绕过的手动步骤，但程序必须：

1. 明确指出当前缺失状态
2. 明确告诉用户应跳转到哪个系统页面
3. 在用户返回后自动重新检测状态
4. 不把 Android 承载器的私有运行目录当成应用可自由管理目录
5. 不把手工编辑配置文件当成正式流程
6. 导入源目录如果部分写入后失败，必须明确提示可基于最近备份执行恢复
7. 目录授权提示必须明确区分“同步快照根目录”和“Android 导入源目录”，不得继续笼统写成“共享目录”
8. 对承载器缺失、输入法未启用和输入法未切换，尽量把用户动作收敛为应用内按钮触发的短链路
9. 非阻断问题输出为显式告警，阻断问题输出为显式报错，仅用于补前置步骤的信息输出为显式提示，并全部写入日志留痕

## 5. 当前仓库规则

开发启动前：

1. `apps/android/` 属于允许继续存在并纳入审计的正式源码根。
2. 与正式主线冲突的演示骨架、重复实现或无效占位页面视为待删除遗留项。
3. 任何重新出现的临时页面、演示页面和脚手架代码都不得反向影响正式需求、架构和共享契约
4. 文档冻结前，禁止创建“先占坑再说”的 Android 工程骨架
5. 进入开发后，应优先在现有 Android 工程基础上继续修改与扩展，不得另起第二套 Android 主线
6. 不允许在 Android 侧重新引入任何跨端共享可执行主线
7. 进入开发阶段后，如需修改本说明文档，也必须遵守冻结例外规则，不得脱离 `docs/` 自行漂移
8. 如果当前任务被明确限定为“只改文档，不改代码”，则目录内现有工程和缓存可以暂时物理保留；但只有完成审计分类后，才允许把可复用实现纳入正式开发基线
