# Windows 端说明

## 头部骨架
- 正式文件路径：`apps/windows/Windows端说明.md`
- 对应的本文档章节：`9.2-Windows 客户端职责`、`14.2-Windows`、`11-图形界面覆盖范围`、`18.2-工程验收`
- 覆盖范围：Windows 目录定位、技术路线、正式职责、强制约束与开发前目录规则。
- 明确非目标：不直接替代 Windows 工程正文、命令面实现或共享契约定义。
- 输入：需求中的 Windows 职责、双入口规则、渲染/DPI 约束、平台编辑边界和目录白名单。
- 输出：Windows 目录允许内容、正式技术路线、职责分层与强制约束说明。
- 真值源：`docs/需求规范.md`、`docs/入口与命令行规范.md` 与 `docs/平台集成规范.md`。
- 正式任务类型：检测、配置、生成、备份、部署、回检、回滚、诊断。
- 正式阶段映射：Windows 目录承接的实现需复用 `detect` 到 `diagnose` 全阶段。
- 失败边界：若重新引入 WPF/XAML、复制主业务逻辑或改写 Android 专属字段，即判定为失效。
- 验收方式：目录说明与 Windows 双入口、渲染主线、平台边界和冻结基线完全一致。

## 1. 目录定位

本目录用于承载 Windows 客户端正式工程。

## 1.1 验证联动

本文档覆盖范围内的正式要求，默认由以下几类逐条验证项共同承接：

1. `doc_baseline-*`
   - 用于验证目录说明、白名单边界、历史存量口径和冻结规则传播。
2. `platform_implementation-windows-*`
   - 用于验证 Windows 技术路线、双入口边界、GDI 主线、目录职责和平台编辑边界。
3. `end_to_end-*`
   - 用于验证 Windows 用户路径、安装/部署/回滚/诊断闭环和高 DPI 可用性。

使用规则固定如下：

1. 本文件新增、删减或改义了 Windows 技术路线、目录边界、双入口规则、安装入口、目录入口、日志入口、高 DPI 约束或历史存量处理口径时，必须同步更新 `docs/需求验证映射.md`、`docs/测试与验收矩阵.md`、`docs/平台集成规范.md`、`docs/入口与命令行规范.md`、`docs/用户手册.md` 与 `docs/需求变更记录.md`。
2. 本文件对应证据默认落到 `/diagnostics/doc-baseline/requirements/`、`/diagnostics/platform-implementation/windows/requirements/` 或 `/diagnostics/end-to-end/requirements/`。
3. 本文件中的目录规则、双入口、GDI / TextRenderer 主线、高 DPI 约束和历史存量限制都必须能在逐条验证动作中点名到具体目录、入口、窗口、诊断对象或平台结果对象。

开发启动前，本目录允许保留现有 Windows 工程文件；这些文件必须先按冻结文档审计分类。只有与正式主线冲突的并行工程、无效壳和遗留路线才应清理。

本目录的内容同样受 [需求规范](../../docs/需求规范.md) 与 [文档治理与冻结规范](../../docs/文档治理与冻结规范.md) 约束。

本说明文档是派生文档，不具备高于 `docs/` 的裁决权。

## 2. 正式技术路线

Windows 客户端固定采用：

1. `C#`
2. `WinForms`
3. `GDI / TextRenderer`
4. `PerMonitorV2`
5. 命令行与图形界面

进入开发后，应继续以现有 Windows 工程为基础修改、更新和增加；只有与正式主线冲突的对象才应删除。

## 3. 正式职责

Windows 客户端必须承担：

1. 检测 Windows 承载器 `Weasel`
2. 检测字体、路径、部署器与 DPI 环境
3. 提供命令行与图形界面双入口
4. 在 Windows 正式工程内部实现应用服务层
5. 读取和保存配置模型
6. 生成并写入 Windows 目标配置，包括 `weasel.custom.yaml`
7. 执行备份
8. 触发部署器
9. 收集日志
10. 参与跨端同步与恢复
11. 回检运行状态
12. 输出结构化诊断结果
13. 尽量提供承载器安装入口、目标目录入口、日志入口与重新检测入口，减少用户自行搜索系统路径的负担

Windows 图形界面编辑边界固定如下：

1. Windows 侧必须编辑全部共享字段与 Windows 专属字段。
2. Android 专属字段在 Windows 侧只允许只读展示、同步结果展示和跳转提示。
3. Windows 保存配置时必须保留 Android 专属字段原值，不得清空、重置或重写。

正式工程启动后，本目录至少应能落出以下职责分层：

1. 命令行入口层
2. 图形界面入口层
3. Windows 应用服务层
4. 平台适配层
5. 与共享契约层的接入层

## 4. 强制约束

1. 文本渲染主线保持 `GDI / TextRenderer`
2. 不引入 WPF
3. 不引入 XAML 运行时
4. 不引入浏览器内核
5. 高 DPI 支持从第一阶段起就是硬约束
6. 命令行与图形界面仅做入口适配，不得复制主业务逻辑
7. Windows 正式工程必须直接消费共享契约层，不得依赖跨端共享可执行核心层
8. 若目标目录部分写入后失败，必须明确提示当前目标状态已被修改，并允许基于最近备份执行回滚
9. 在普通消费者应用边界内，Windows 端必须尽量把安装、部署、日志查看和修复动作收敛为应用内按钮和明确引导
10. 非阻断问题输出为显式告警，阻断问题输出为显式报错，仅用于补前置步骤的信息输出为显式提示，并全部写入日志留痕

## 5. 当前仓库规则

开发启动前，当前目录属于允许继续存在并纳入审计的正式源码根。

任何后续重新出现的以下内容，都属于违背基线的遗留回潮：

1. `.xaml`
2. `.xaml.cs`
3. `UseWPF`
4. 临时 WinForms 演示壳
5. 与 Windows 应用服务层脱节的独立命令行或独立图形界面
6. 与正式主线冲突的并行工程骨架
7. 进入开发阶段后脱离冻结基线自行追加的目录规则
8. 如果当前任务被明确限定为“只改文档，不改代码”，则目录内现有工程和缓存可以暂时物理保留；但只有完成审计分类后，才允许把可复用实现纳入正式开发基线
